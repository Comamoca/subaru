*** 問題
CLAUDE.md に「NEVER use simulation」と明記されているにもかかわらず、
=simulateJavaScriptExecution= が実行失敗時のフォールバックとして残っている。

フォールバック内で正規表現を使って生成済みJSコードをパースし出力を「推測」する手法は
根本的に誤っており、誤った結果を返す可能性がある。

関連メソッド (すべて削除対象):
- =simulateJavaScriptExecution= (L689)
- =processEchoCalls= (L714)
- =processPrintlnCalls= (L747)
- =handleHttpResponseEcho= (L736)
- =extractUrlFromCode= (L742)
- =formatEchoValue= (L796)
- =simulateHttpFetch= (L765)
- =escapeForJsonString= (L787)
- 定数 =FALLBACK_URL= (L52), =ECHO_CALL_PATTERN=, =PRINTLN_PATTERN=, =URL_EXTRACTION_PATTERN= (L55-57)

*** 対応
=executeJavaScript= メソッドを単純化し、実行失敗時はエラーをそのまま上位に伝播させる。

#+BEGIN_SRC typescript
private async executeJavaScript(
  jsCode: string,
  moduleName: string = "main",
  allModules?: Map<string, string>,
  ffiFiles?: Map<string, string>,
): Promise<RunResult> {
  return await this.realJavaScriptExecution(jsCode, moduleName, [], allModules, ffiFiles);
}
#+END_SRC

** A2: WASMモジュール管理クラスの分離
:PROPERTIES:
:FILE: src/gleam_runner.ts → src/wasm/wasm_module.ts (新規)
:LINES: 1-140, 181-252, 277-452
:PRIORITY: 高

*** 問題
=GleamRunner= クラス (1074行) が以下の責務をすべて持っている:
- WASMコンパイラのパス解決・ダウンロード
- WASMモジュールの初期化
- コンパイル処理
- 実行処理
- stdlib読み込み
- preloadスクリプト管理
- ログ/警告制御

単一責任原則に違反しており、テストが困難。

*** 問題箇所
=compile= メソッド内で毎回動的importを3回実行している (L292, L319, L343):

#+BEGIN_SRC typescript
// 毎回のcompile呼び出しで重複実行される
const { reset_filesystem, write_module, ... } = await import(`file://${this.wasmPath}/gleam_wasm.js`);
// ...
const { write_file } = await import(`file://${this.wasmPath}/gleam_wasm.js`);
// ...
const { read_file_bytes } = await import(`file://${this.wasmPath}/gleam_wasm.js`);
#+END_SRC

*** 対応
新ファイル =src/wasm/wasm_module.ts= を作成し =WasmModule= クラスを定義:

#+BEGIN_SRC typescript
// src/wasm/wasm_module.ts
export class WasmModule {
  private api: WasmApi; // initialize()時に一度だけロード

  async initialize(wasmPath: string): Promise<void>
  resetFilesystem(projectId: number): void
  writeModule(projectId: number, name: string, code: string): void
  compile(projectId: number, target: "javascript" | "erlang"): void
  readCompiledJavaScript(projectId: number, name: string): string | undefined
  popWarning(projectId: number): string | undefined
}
#+END_SRC

=GleamRunner= はこれを利用するだけにし、動的importを排除する。

=setupLogging= がグローバルの =console.log= を上書きしている:

* Track C: Worker改善

** C1: ファイルベース出力キャプチャの廃止
:PROPERTIES:
:FILE: src/worker/execution_worker.ts
:LINES: 62-70, 239-247
:PRIORITY: 中
:END:

*** 問題
現在の出力キャプチャは一時ファイル経由で行われている:

#+BEGIN_SRC typescript
// worker内でファイルに書き込み
const __outputPath = "${outputFile}";
function __captureOutput(msg) {
  Deno.writeTextFileSync(__outputPath, line + "\n", { append: true });
}
// ...
// 実行後にファイルを読む
const fileOutput = await Deno.readTextFile(outputFile);
#+END_SRC

この手法は:
- ディスクI/Oが不必要に発生する
- ファイルパスをJS文字列に埋め込む (インジェクションリスク)
- =gleam_runner.ts= 側でtempDirの作成・削除が必要

この手法を置き換える =postMessage= 版の実装は =gleam_runner.ts= の =getModuleStubs().gleamIo=
(L660-672) に定義されているが、worker は受け取った =moduleStubs.gleamIo= を使っておらず、
自前のファイルベース =gleamStdlibStub= (L61-69) で上書きしているため実際には使われていない。

*** 対応
=execution_worker.ts= のハードコードされた =gleamStdlibStub= を削除し、
=moduleStubs.gleamIo= を =gleam_stdlib.mjs= として書き込む形に切り替える。
これにより =print= / =println= が =self.postMessage({ type: "output", line })= を
経由するようになり、tempFileとoutputFileを廃止できる。
Worker側のtempDirはモジュール書き込み用途のみに縮小 (→ C2でさらに改善)。

** C2: Worker内モジュール書き込みの最適化
:PROPERTIES:
:FILE: src/worker/execution_worker.ts
:LINES: 200-231
:PRIORITY: 低
:END:

*** 問題
コンパイル済みモジュールを毎回tempDirにファイルとして書き込んでから
=import()= で読み込んでいる。大量モジュール (stdlib含む) で多数のファイルI/Oが発生。

*** 対応
Data URLを使ったインメモリimportを検討:

#+BEGIN_SRC typescript
// ファイルに書かずにData URLでimport
const blob = new Blob([moduleCode], { type: "text/javascript" });
const url = URL.createObjectURL(blob);
const mod = await import(url);
#+END_SRC

ただしDenoのWorker環境でのBlob URL対応状況を確認した上で実施する。
対応していない場合はファイルベースを維持。

** D1: resolveWasmPath の優先順序修正
:PROPERTIES:
:FILE: src/gleam_runner.ts
:LINES: 93-139
:PRIORITY: 低
:END:

*** 問題
コメントには「configPath → ./wasm-compiler → cache → auto-download」と書かれているが
実際のコードは「configPath → cache → local → auto-download」の順で動いている。

#+BEGIN_SRC typescript
// コメント: Priority: configPath -> ./wasm-compiler -> cache -> auto-download
// 実装: configPath -> cache -> localPath -> auto-download  ← 不一致
const cachePath = getWasmCacheDir();
if (await wasmExists(cachePath)) { return cachePath; }     // cacheが先

const localPath = `${Deno.cwd()}/wasm-compiler`;
if (await wasmExists(localPath)) { return localPath; }     // localが後
#+END_SRC

*** 対応
コードとコメントを一致させる。localPathを先にチェックする方が自然
(ユーザーが明示的に配置したファイルを優先すべき)ため、コードを修正:

#+BEGIN_SRC typescript
// 2. Check local ./wasm-compiler directory (user-placed, takes priority over cache)
// 3. Check cache directory
#+END_SRC

*** 問題
stdlib読み込み後に =test_io= という無意味なモジュールを書き込んで
gleam/ioが使えるかをテストしている:

#+BEGIN_SRC typescript
try {
  const testCode = 'import gleam/io\npub fn check() { io.println("check") }';
  trackingWriteModule(this.projectId, "test_io", testCode);  // 副作用あり

=StdlibLoader= が =LoadResult.errors= を返すため、それを使ってロード成否を判定する。
フォールバックio追加は =StdlibLoader.addFallbackIo= を明示的に呼ぶ形に変更。

*** 問題

#+BEGIN_SRC typescript
} catch (error) {
